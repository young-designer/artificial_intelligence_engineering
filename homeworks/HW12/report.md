# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- `S12-hw-dataset.csv` — временной ряд с почасовыми данными за период с 01.01.2025 по 29.06.2025.
- Данные разделены хронологически 70%/15%/15% (train/val/test) без перемешивания, чтобы сохранить временной порядок и избежать утечки из будущего.
- B1 (naive-last), B2 (moving-average), B3 (Ridge на лаговых признаках), R1 (GRU на оконном представлении).
- Лучшим стал R1 (GRU) с val_MAE=5.88, так как рекуррентная архитектура лучше улавливает сложные временные зависимости, чем простые статистические baseline-модели.

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: target
- Частота временного ряда: 1 час
- Горизонт прогноза: 1 шаг (1 час)
- Размер датасета: 4320 наблюдений
- Диапазон дат: 2025-01-01 — 2025-06-29
- Есть ли дополнительные внешние признаки: Нет (использовались только производные от target и date)
- Комментарий (2-4 предложения): Временной ряд показывает явный тренд (среднее значение растёт со временем) и вероятную суточную сезонность (периодические пики). Выбросы присутствуют (min = 69.1, max = 210.1), что требует осторожности при масштабировании. Ряд нестационарен, что подтверждается визуальным анализом графика.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: Последовательное разделение по времени без перемешивания (iloc[:train_size], iloc[train_size:train_size+val_size], iloc[train_size+val_size:]).
- Какие доли или интервалы использовались: 70% train / 15% validation / 15% test.
- Почему `random split` здесь некорректен: Нарушается временная зависимость между наблюдениями, data leakage, не оценивается способность модели прогнозировать будущие значения

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: lag_1, lag_7, lag_14
- Какие rolling-признаки использовались: rolling_mean_7, rolling_std_7
- Какие календарные признаки использовались: hour, day_of_week, day_of_month, month, is_weekend
- Как обрабатывались пропуски после построения лагов/rolling: Строки с NaN удалены после построения лагов.
- Использовалось ли масштабирование и где именно оно обучалось: StandardScaler, обучен только на train.

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): Прогноз равен последнему известному значению из train
- B2 (`moving-average`): Прогноз равен скользящему среднему за последние 7 часов
- B3 (`ridge-lag-features`): Ridge Regression (alpha=1.0) на лаговых, rolling и календарных признаках
- R1 (`gru-forecast`): GRU (hidden_size=64, num_layers=2, dropout=0.2) с window_size=24, optimizer=Adam, lr=0.001

Дополнительно:

- Основная метрика выбора лучшей модели: best_val_mae
- Дополнительные метрики: RMSE, MAPE
- `window_size` для `GRU`: 24
- Batch size: 32
- Максимальное число эпох: 50
- Optimizer: Adam
- Learning rate: 0.001
- Как сохранялась лучшая модель: По минимальному val_loss с Early Stopping (patience=10)

## 7. Анализ

(8-15 предложений)

Нужно прокомментировать:

- B1 (Naive Last) показал худший результат, так как простое копирование последнего значения не учитывает тренд и сезонность ряда. Временной ряд имеет выраженную динамику, поэтому этот прогноз сильно отклоняется от фактических значений.
- B2 (Moving Average) улучшил результат до MAE ≈ 14.8, усредняя последние 7 значений. Однако B3 (Ridge) с лаговыми признаками показал значительный прогресс (MAE ≈ 6.4), так как модель научилась взвешивать различные лаги и календарные признаки оптимальным образом.
- Ridge Regression использует регуляризацию, что предотвращает переобучение на множестве признаков. Модель смогла выявить, что lag_1 имеет наибольший вес, а также учесть влияние дня недели и месяца.
- GRU способна запоминать долгосрочные зависимости благодаря механизму гейтов. Window_size = 24 позволяет модели учитывать суточный цикл, что критично для почасовых данных. Early Stopping предотвратил переобучение.
- Тренд явно присутствует (рост среднего значения со временем). Сезонность прослеживается на графике остатков — есть периодические паттерны ошибок, что указывает на не полностью уловленную суточную/недельную сезонность.
- Потенциальные источники утечки данных: масштабирование на всём датасете вместо только train — предотвращено; использование будущих значений при построении лагов — предотвращено через shift; перемешивание данных перед split — предотвращено через temporal split.
- Выбор модели по валидации предотвращает «подгонку» под тестовые данные и сохраняет test как независимый, объективный финальный экзамен; если выбирать по test, мы неявно переобучаемся на него, и его метрики перестают отражать реальную способность модели обобщать на новые, невиданные данные.
- Наибольшие ошибки наблюдаются на резких скачках значения target (выбросы). Остатки показывают, что модель систематически недооценивает пики и переоценивает спады, что типично для регрессионных моделей.

## 8. Итоговый вывод

(3-7 предложений)

- Для быстрой оценки и продакшена я бы выбрал B3 (Ridge) — он даёт хорошее качество (MAE ≈ 6.0), быстро обучается, интерпретируем и не требует тонкой настройки гиперпараметров.
- Если данные имеют простую структуру, мало шума, и требуется быстрое развёртывание — лаговые признаки + линейная модель достаточно эффективны.
- При наличии сложных нелинейных зависимостей, долгосрочных паттернов, большом объёме данных и требованиях к максимальной точности. В данном случае GRU дала улучшение на ~7.6%, что может быть существенно для бизнес-задач.
- Ключевой принцип — это никакого перемешивания. Temporal split обязателен, чтобы оценка на validation/test отражала реальную способность модели прогнозировать будущее. Test используется только один раз для финальной проверки лучшей модели, выбранной по validation.
